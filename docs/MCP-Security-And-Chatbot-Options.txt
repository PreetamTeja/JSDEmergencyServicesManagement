JSD TATA Emergency Services Platform
Prepared: 30 June 2026


SECTION 1. WHAT MCP IS AND HOW IT WORKS IN OUR SYSTEM.

MCP stands for Model Context Protocol. It is an open standard that allows AI assistants to call external tools in a structured and controlled way. Instead of the AI guessing at information or generating it from memory, it can invoke a defined tool, receive a real result, and use that result to answer accurately.

In our system, the MCP server is a Node.js script located at infra/mcp/cloudwatch-server.mjs. It is not a web server that runs permanently. It is a process that Claude Code, the AI development tool, spawns on demand when a developer asks a question that requires live infrastructure data. The process communicates with Claude Code through standard input and standard output, which is a pipe between two processes on the same machine. There is no open network port, no daemon running in the background, and no persistent connection of any kind.

The registration of this server is defined in a file called .mcp.json at the root of the project. This file is not committed to the repository because it contains the MCP API key. It is listed in .gitignore and must be created locally by each developer. The file tells Claude Code where to find the server script, what command to run it with, and what environment variables to pass, specifically the backend API base URL and the MCP API key.

When a developer opens Claude Code and asks a question such as how many errors occurred in the last hour, Claude Code reads the .mcp.json registration, spawns the MCP process, sends a JSON request through the pipe, receives the JSON response, and then uses that data to form a human-readable answer. The entire interaction happens locally on the developer machine.

The five tools exposed by our MCP server are get_lambda_metrics, get_recent_errors, query_logs, get_infra_summary, and get_infra_metrics. Each tool makes a single HTTP call to our backend API at the /infra/metrics endpoint, authenticated using the MCP API key. The MCP server does not call AWS directly and requires no AWS credentials on the developer machine.

The system has been verified working. A live test on 30 June 2026 returned the following results through the full MCP chain: status healthy, 1989 invocations in the last hour, error rate zero percent, average duration 166 milliseconds, zero cold starts, and zero Lambda errors.


SECTION 2. AUTHENTICATION: THE MCP API KEY.

The MCP server authenticates using a dedicated API key with the scope MCP. This is the same API key system already used by the hospital integration team and the fuel team. The key is sent in the x-api-key HTTP header on every request to the /infra/metrics endpoint.

The MCP key has three properties that make it suitable for developer tooling. First, it does not expire. Unlike a Cognito JWT, which expires after one hour and requires the developer to retrieve a fresh token from the browser session each time, the MCP key is permanent until manually rotated. Second, it is narrowly scoped. The backend only permits the MCP scope to access the /infra endpoint. It cannot call any emergency, fleet, booking, or employee endpoint. Third, it is stored only in the Lambda environment variables and in the developer's local .mcp.json file, neither of which is ever committed to version control.

The key is stored in the Lambda environment variable API_KEYS as part of a JSON map where the key value maps to the string MCP. The backend reads this map on every request, extracts the x-api-key header, looks it up in the map, and checks whether the resulting scope is permitted to access the requested path. If the key is absent, invalid, or has the wrong scope, the backend returns 403 Forbidden before any CloudWatch query is made.

To rotate the key, generate a new 32-character hex string, update the API_KEYS environment variable on the Lambda function, and update the MCP_API_KEY value in each developer's local .mcp.json file. The old key stops working immediately once the Lambda environment is updated.


SECTION 3. THE FULL REQUEST CHAIN.

A tool call from Claude Code follows this exact path.

Claude Code spawns the MCP server process using the command and arguments defined in .mcp.json. The process starts and reads API_BASE_URL and MCP_API_KEY from the environment variables that Claude Code injected. Claude Code sends a JSON-RPC 2.0 request through stdin, for example a call to get_infra_summary with no arguments.

The MCP server receives the request, selects the appropriate function, and calls the backend API at /infra/metrics with the query parameters range_min and period_min set to 60, meaning the last hour. The request includes the header x-api-key with the MCP key value.

The Lambda function receives the request at the API Gateway endpoint in eu-west-1. It extracts the x-api-key header, looks up the value in the API_KEYS map, and finds the scope MCP. It checks that MCP is permitted to access the infra path, which it is. It then queries CloudWatch using the Lambda's own IAM role, which has read-only CloudWatch permissions, and queries the CloudWatch Logs group for recent error lines. It assembles the result and returns JSON.

The MCP server receives the JSON, formats it into an MCP content block, and writes the response to stdout. Claude Code reads the response from stdout and presents the information to the developer in plain language.

The entire chain from developer question to answer takes approximately two to three seconds.


SECTION 4. WHAT THE MCP SERVER CAN AND CANNOT DO.

The MCP server can retrieve the following data. It can get Lambda invocation counts, error counts, throttle counts, average duration, p99 duration, cold start counts, and maximum concurrent executions for the transport API Lambda function. It can get recent ERROR and WARN level log lines from the Lambda's CloudWatch log group. It can filter those log lines by a keyword pattern. It can return a combined health summary including a status field that is either healthy or degraded, a list of active issues such as high error rate or elevated p99 latency, and the key metric values for the last hour.

The MCP server cannot do any of the following. It cannot write to DynamoDB, read from DynamoDB, or access any patient, employee, fleet, or booking data. It cannot invoke any other Lambda function. It cannot modify API Gateway, IAM roles, S3 buckets, or any other AWS resource. It cannot create, update, or delete CloudWatch alarms. It cannot call any backend endpoint other than /infra/metrics. It cannot accept any free-form command from the developer and execute it. Every tool has a fixed set of typed parameters and makes a single predetermined HTTP call. There is no shell access, no eval, and no dynamic code execution of any kind.


SECTION 5. WHY NO AWS CREDENTIALS ARE NEEDED ON THE DEVELOPER MACHINE.

The original design of the MCP server called AWS CloudWatch directly using the AWS SDK. This required developers to have AWS credentials configured locally, typically through the AWS CLI. In our environment, developers do not have individual AWS CLI access.

Rather than granting AWS credentials to each developer, we redesigned the MCP server to proxy through the backend API. The backend Lambda already has the necessary CloudWatch permissions through its IAM role, specifically the actions cloudwatch:GetMetricData, cloudwatch:GetMetricStatistics, logs:FilterLogEvents, logs:GetLogEvents, and logs:DescribeLogGroups. All of these are read-only.

By routing requests through the Lambda, we achieve the following. No AWS credentials are ever stored on a developer machine. The CloudWatch access is always mediated by the Lambda's own narrowly scoped IAM role. The MCP key provides a clear audit trail since the Lambda logs every request. If we ever need to add rate limiting, logging, or additional data to the infra endpoint, we do it in one place and all clients including the React dashboard and the MCP server benefit automatically.


SECTION 6. RISKS AND HOW THEY ARE MANAGED.

There are two residual risks that are accepted and managed.

The first is MCP key compromise. If a developer's local .mcp.json file were accessed by an attacker, the MCP key could be extracted. The consequence is that the attacker could read CloudWatch metrics and recent log lines for the Lambda function. They could not access any user data, modify any resource, or perform any destructive action. The key can be rotated immediately by updating the Lambda environment variable, which invalidates it within seconds. The key never appears in version control because .mcp.json is gitignored.

The second is prompt injection through log content. CloudWatch log lines are returned verbatim to Claude Code. A malicious actor who could write arbitrary content to the Lambda log group could craft a log message containing instructions intended to influence the AI assistant's behaviour. This risk is low because only the Lambda function writes to its own log group, which means an attacker would need to first compromise the Lambda itself. Additionally, each log line is truncated at 500 characters, and Claude Code treats tool results as data rather than instructions.


SECTION 7. OPTION 2. A DIRECT CHATBOT IN THE ADMIN INFRA HEALTH PAGE.

Rather than using the MCP server through Claude Code, we can embed an AI chatbot directly into the Infra Health page of the React application at the /admin/infra route. This chatbot would sit alongside the existing KPI strip, charts, and error log.

The chatbot works as follows. The operator types a question in plain English into a chat input, for example, why are there so many errors this afternoon, or is the error rate trending up, or show me all timeout errors from the last two hours. The React frontend sends this question along with the conversation history to a new backend endpoint at POST /infra/chat. The Lambda handler fetches the current CloudWatch metrics and recent log lines, assembles them as context alongside the operator's question, and calls Amazon Bedrock Nova Lite, which is the same model already used by the voice agent, using the eu-west-1 region endpoint. Bedrock returns a plain English answer. The Lambda sends the answer back to the frontend, where it appears in the chat panel below the question.

This approach has several advantages over the MCP approach. It requires no local setup. The operator does not need to install anything, configure any files, or restart any tool. It works from any browser on any machine, including a mobile device or a colleague's workstation. It is accessible to operations managers and IT administrators who need to understand the platform's health without using a developer tool. Authentication uses the same Cognito admin JWT already in the browser session, so there is no additional login step. The chatbot can support follow-up questions within a session by retaining the conversation history in the browser and sending it with each request.

The limitation of this approach compared to the MCP approach is that the chatbot is scoped to the data the backend already provides at /infra/metrics. The MCP server, when used through Claude Code, allows the AI to work with the full repository including code, configuration, and git history, alongside the CloudWatch data. For debugging a specific bug in the code while simultaneously checking whether it caused an error spike in production, the MCP approach gives a more complete picture. For operations monitoring and incident response by team members who are not using Claude Code, the embedded chatbot is the better experience.

The two options are not mutually exclusive. Both can be present simultaneously, with the MCP server serving developers through Claude Code and the embedded chatbot serving the wider operations team through the browser.

If Option 2 is chosen, four changes are required. First, a new endpoint POST /infra/chat is added to handler.mjs. This endpoint is gated on the admin JWT, accepts a messages array and an optional data_window_min parameter, fetches the current CloudWatch metrics, and calls Bedrock Nova Lite with the assembled context. Second, the Lambda IAM role receives a new inline policy granting bedrock:InvokeModel for the Nova Lite model ARN in eu-west-1. Third, a ChatPanel component is added to InfraHealthPage.jsx, consisting of a scrollable message list, a text input, and a send button. Fourth, a new api.chatInfra method is added to services/api.js. No new AWS services, infrastructure, or databases are required.


SECTION 8. SUMMARY.

The MCP integration is production-ready and has been tested live. The key does not expire. No AWS credentials are needed on the developer machine. The scope is read-only and limited to infrastructure metrics. The key is never committed to version control. Rotation takes under one minute. The embedded chatbot option described in Section 7 can be built on top of the existing Lambda and Bedrock infrastructure without any new AWS services.
